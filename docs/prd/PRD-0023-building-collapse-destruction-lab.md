# PRD-0023 Building Collapse Destruction Lab

## Vision

`v33` 的目标不是立刻把整座 `70km x 70km` 城市改造成“全图任意建筑 fully physical destruction”的重系统，而是先把一条**低风险、强反馈、可快速 `F6` 验收**的建筑坍塌玩法主链做通：玩家在一个独立实验场景中，使用现有火箭弹武器持续轰击一栋楼；建筑能够识别命中的楼体与命中点，维护正式的建筑生命值；血量进入中度受损阶段后，在命中点附近生成低成本裂纹/受损视觉；同时在后台准备该建筑的预碎裂坍塌数据；当血量进入濒毁阈值后，原建筑主体切换为预准备好的可碎裂坍塌版本，在重力作用下分块倒塌，最终只留下底座/少量残骸，并在定时窗口后清理大部分碎块。

`v33` 的关键不是“某一栋楼看起来炸得很热闹”，而是把这条链路拆成两个工程阶段并各自交付完整。第一阶段在一个可独立运行的实验世界中完成闭环：平直大地、玩家、现有控制/武器/战斗系统、环境、目标建筑、伤害状态机、裂纹、预碎裂、临界替换、坍塌、碎块清理。这一阶段强调**验证快、调试快、`F6` 就能看效果**。第二阶段才把同样的 runtime 逻辑接回主世界建筑体系：通过现有 `building_id + inspection_payload + chunk renderer` 主链，给主世界建筑叠加同样的伤害与坍塌 contract，但首版不要求持久化，也不要求离开游戏后保留建筑被毁状态。

## Background

- 当前仓库已经有稳定的玩家战斗主链：`PlayerController -> CityPrototype -> projectile / grenade / missile`。
- 当前仓库已经有正式的建筑识别基础：
  - `CityWorldInspectionResolver.gd` 能从命中结果中解析建筑 payload 与 `world_position`
  - `CityChunkProfileBuilder.gd` 为建筑生成稳定 `building_id`
  - `CityChunkScene.gd` / `CityChunkRenderer.gd` 能按 `building_id` 获取建筑 contract 或定位 override node
  - `CityBuildingSceneBuilder.gd` 当前主世界默认建筑本质上是 `StaticBody3D + BoxMesh / BoxShape3D` 的盒子式建筑
- 用户已明确冻结 `v33` 的研发路径：
  - 先做独立实验场景，完成 1-5 步玩法验证
  - 再把相同逻辑移植到主世界
  - 首版不持久化建筑摧毁结果；离开/重开后建筑恢复
- 用户建议的破坏链路已冻结为：
  1. 城市现状不改
  2. 建筑能识别伤害与命中点
  3. 进入中度受损时显示低成本裂纹
  4. 在后台准备预碎裂数据/模型
  5. 在濒毁阈值时替换为可碎裂坍塌体并完成清理

## Scope

本 PRD 只覆盖 `v33 building collapse destruction lab`。

包含：

- 新增一个可 `F6` 独立运行的建筑坍塌实验场景
- 实验场景中接入玩家、现有武器/战斗系统、平直地面、环境和目标建筑
- 首版实验目标建筑可以直接复用：
  - `res://city_game/serviceability/buildings/generated/bld_v15-building-id-1_seed424242_chunk_137_136_001/building_scene.tscn`
- 建筑运行时生命值 / 命中点 / 状态机
- 中度受损裂纹/受损视觉
- 后台预碎裂准备
- 濒毁阈值触发替换为可碎裂坍塌体
- 坍塌后碎块清理与底座残留
- 第二阶段把同一 runtime 逻辑扩展到主世界建筑链

不包含：

- 首版不做建筑摧毁结果持久化
- 首版不做“全图所有楼都同时 ready-to-collapse”
- 首版不做任意 mesh 实时布尔碎裂
- 首版不做 Blast / APEX / PhysX / 外部 C++ 破坏 SDK 接入
- 首版不做玻璃、室内装修、家具等二级破坏
- 首版不要求所有建筑 archetype 都拥有不同坍塌风格

## Non-Goals

- `v33` 不追求严格结构力学仿真
- `v33` 不追求建筑碎块永久保留
- `v33` 不追求整城同时爆破的极端场景
- `v33` 不追求 runtime 实时从任意复杂 mesh 精确切碎

## Requirements

### REQ-0023-001 必须先交付一个独立的建筑坍塌实验场景

**动机**：主世界流式城市验证成本高，先要有一个 `F6` 即可运行的快速实验场，验证完整坍塌玩法链。

**范围**：

- 新增一个独立实验场景，场景负责静态摆放：
  - 平直大地
  - 环境/光照
  - 玩家
  - 现有武器/战斗系统所需根节点
  - 目标建筑
- 目标建筑首版可直接复用 `building_scene.tscn`
- 实验场景必须场景文件持有静态摆放；GDScript 只负责 runtime 逻辑

**非目标**：

- 不要求实验场景复刻主世界全部 streaming 细节

**验收口径**：

- `F6` 运行实验场景即可进入可操作战斗环境。
- 玩家能在实验场景中使用正式武器链攻击目标建筑。
- 反作弊条款：不得把楼、地面、玩家位置全都用 `Node3D.new()` 在脚本里临时搭出来替代正式实验场景。

### REQ-0023-002 建筑必须能维护正式伤害状态，并识别命中点

**动机**：没有正式 `building_id + hit_position + health` contract，就无法支撑裂纹、预碎裂和坍塌触发。

**范围**：

- 目标建筑必须维护：
  - `building_id`
  - `max_health`
  - `current_health`
  - `last_hit_world_position`
  - `last_hit_local_position`
  - `damage_state`
- 玩家火箭弹命中建筑时，必须把伤害施加到该建筑 runtime，而不是只消耗导弹本身
- 状态机至少包括：
  - `intact`
  - `damaged`
  - `fracture_preparing`
  - `collapse_ready`
  - `collapsing`
  - `collapsed`

**非目标**：

- 首版不要求楼与楼之间伤害传播

**验收口径**：

- 自动化测试至少断言：命中目标建筑后，`current_health` 会减少，且记录命中点。
- 自动化测试至少断言：中度受损阈值与濒毁阈值能稳定触发状态迁移。
- 反作弊条款：不得通过“导弹命中后直接把楼删掉”来伪装建筑伤害系统已存在。

### REQ-0023-003 建筑在中度受损时必须显示低成本裂纹/受损视觉

**动机**：用户明确希望在未坍塌前先看到持续轰击造成的可见伤痕，而不是一直完好到瞬间替换。

**范围**：

- 当血量低于中度受损阈值时，在命中点附近显示裂纹、破损或焦痕视觉
- 视觉首版优先低成本技术：
  - `Decal`
  - crack quad / mesh
  - 或 shader mask 叠加
- 裂纹视觉应与命中点位置相关，而不是固定贴在楼正面中心

**非目标**：

- 首版不要求多层累积裂纹的复杂材质系统

**验收口径**：

- 自动化测试至少断言：进入中度受损状态后，裂纹节点/状态存在。
- 手测/实验场景验收必须能看到裂纹出现在受击区域附近。
- 反作弊条款：不得只改楼体整体颜色或统一换材质来冒充命中点裂纹。

### REQ-0023-004 中度受损后必须在后台准备预碎裂坍塌数据

**动机**：用户希望“先受损，再后台准备，最后临界时瞬时切换坍塌”，避免临界一刻现算造成卡顿。

**范围**：

- 建筑进入中度受损后，启动预碎裂准备流程
- 后台流程首版允许只准备“坍塌 recipe / chunk 布局 / 块数据”，不要求直接在后台改场景树
- 预碎裂结果必须体现：
  - 从命中区域开始的局部破坏倾向
  - 建筑主体是中空壳体，而不是纯实心石块
  - 有限数量的坍塌大块（首版以可控数量为主）

**非目标**：

- 首版不要求 runtime 对任意复杂 authored mesh 做高精度布尔碎裂

**验收口径**：

- 自动化测试至少断言：建筑进入 `fracture_preparing` 后会产生可复用的 fracture prepared state。
- 自动化测试至少断言：未进入濒毁状态前，不会提前替换掉原建筑主体。
- 反作弊条款：不得把“临界时现算碎块”伪装成预碎裂准备完成。

### REQ-0023-005 建筑在濒毁阈值时必须切换为可碎裂坍塌体，并完成清理

**动机**：这是一整套玩法链的终点，也是用户要的“最终坍塌”效果。

**范围**：

- 当建筑血量低于濒毁阈值时：
  - 原建筑主体隐藏或移除
  - 替换成预准备好的可碎裂坍塌体
  - 在重力作用下完成分块坍塌
- 坍塌后必须：
  - 保留底座或少量残骸
  - 定时清理大部分碎块
- 首版不持久化摧毁结果；重新进入世界后建筑恢复

**非目标**：

- 首版不要求碎块永久堆积

**验收口径**：

- 自动化测试至少断言：濒毁阈值触发后，建筑会进入 `collapsing` / `collapsed` 状态。
- 手测必须能看到原建筑被可碎裂坍塌体替换并坍塌。
- 自动化测试至少断言：碎块会在清理窗口后减少，只保留底座/少量残骸。
- 反作弊条款：不得只播放一个倒塌动画或直接把楼隐藏后留一地粒子来宣称“坍塌已完成”。

### REQ-0023-006 第二阶段必须把实验世界 runtime 扩展到主世界建筑链

**动机**：实验场不是最终交付，必须确保实验验证过的逻辑能接回主世界。

**范围**：

- 把建筑伤害 / 裂纹 / 预碎裂 / 坍塌 runtime 扩展到主世界建筑
- 优先接入近景主世界建筑；首版不要求远景 HLOD 楼体同步坍塌
- 主世界仍保持“内存态破坏，不持久化”

**非目标**：

- 首版不要求离开 chunk 再回来仍保留坍塌结果

**验收口径**：

- 自动化测试至少断言：主世界中可识别的建筑命中后会进入正式伤害链。
- 至少一条主世界 E2E / focused world test 证明：玩家可在主世界中把一栋楼打入坍塌链。
- 反作弊条款：不得只在实验场景生效而把主世界范围永久留在计划外。

## Success Metrics

- `F6` 运行独立实验场即可完整验收建筑伤害到坍塌的 1-5 步
- 建筑中度受损后能看到受击裂纹
- 建筑濒毁时能替换为可碎裂坍塌体并完成清理
- 同一套 runtime 能在第二阶段接入主世界近景建筑
